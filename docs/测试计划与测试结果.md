## 1 引言

### 1.1 编写目的

本文档是“万花筒高并发在线票务服务系统”的测试计划与测试结果说明书。编写本文档的目的在于，对系统测试目标、测试范围、测试环境、测试用例、执行结果和遗留风险进行完整记录，为项目验收、展示评审和后续维护提供可追溯依据。

本文档面向的读者包括：项目评审人员、系统开发人员、测试人员和后续维护人员。评审人员可通过本文档了解系统已验证能力和边界条件；开发与测试人员可依据本文档复现测试过程、定位风险项，并继续补充集成测试和性能测试覆盖。

## 2 测试概要

### 2.1 测试目标

本轮测试围绕系统展示和项目验收所需的核心质量目标展开，重点验证以下内容。

第一，验证项目具备可重复构建能力。后端应能够在 JDK 17 环境下完成 Maven 编译，前端应能够完成 Vite 生产构建并生成可部署产物。

第二，验证核心公共契约和关键业务算法的稳定性。统一响应结构、业务码解析、订单版本解析、字符串工具和自动选座算法应有可重复运行的单元测试覆盖。

第三，验证核心下单链路在高并发条件下能够守住库存一致性。对下单 V4 接口进行 500 并发线程 JMeter 压测，重点统计售空前真实成交请求的吞吐、延迟和超卖情况。

第四，识别当前测试覆盖不足和系统剩余风险，为后续集成测试、故障注入测试和生产级性能优化提供依据。

### 2.2 测试策略

测试策略采用“由低到高、由确定性到场景化”的分层验证方式。

| 层级 | 测试方式 | 验证重点 |
| --- | --- | --- |
| 构建层 | Maven、npm、Docker build | 代码可编译、产物可构建、镜像构建路径可用 |
| 单元层 | JUnit | 工具类、枚举映射、自动选座算法的确定性行为 |
| 编排层 | Docker Compose config | 容器编排文件语法和服务依赖配置 |
| 性能层 | JMeter | 下单 V4 接口在高并发抢票场景下的吞吐、延迟和一致性 |
| 集成层 | 接口链路测试 | 登录、节目、锁座、订单、支付和取消流程闭环 |
| 故障层 | 故障注入 | Redis、Kafka、MySQL 异常时的降级、补偿和恢复 |

### 6.3 性能测试结果

#### 6.3.1 原始数据口径

JMeter 结果文件 `docs/jmeter_result2.jtl.gz` 中存在主样本和 subresult 两类记录。统计性能结果时仅采用 13 列主样本，避免重复计数。

| 项目 | 数值 | 说明 |
| --- | --- | --- |
| JTL 原始行 | 145,858 | 13 列主样本 + 17 列 subresult |
| 有效主样本 | 72,929 | 与 JMeter 主采样口径一致 |
| `195B` 响应 | 1,500 | 成交/锁座成功，数量等于库存 |

| HTTP 错误率 | 0% | JMeter 层面 `success=true` |

由于本轮 JMeter 配置未保存响应体，业务结果通过响应大小、出现时间和库存数量进行分类。`195B` 响应数量恰好等于总库存 1500，且 `200B` 响应在售空边界后大量出现，因此该分类可用于售空前性能统计。后续测试建议把业务 `code`、`message` 和 `orderNumber` 是否存在写入 JMeter sample variable。

#### 6.3.4 结果评价

本轮压测验证了下单 V4 链路在 500 并发线程、1500 座库存条件下能够守住不超卖约束。真实成交响应数量等于库存数量，未观察到成交数超过库存的情况。

从性能表现看，售空前 P50 为 758ms，P95 为 1.743s，P99 为 2.249s，最大响应时间为 2.697s。该结果说明在单机 Docker Compose、单 Redis 实例和 3 个 Program 实例条件下，系统可以完成高并发抢票场景的核心一致性验证，但长尾延迟仍受到 Redis Lua 热点路径和并发爬坡影响。

售空后 `200B` 拒绝请求数量较大，说明测试在库存耗尽后仍持续施压。该类请求适合用于观察拒绝路径和服务反压，不应混入售空前真实成交请求的延迟和吞吐统计。

## 7 问题分析与风险

### 7.1 已确认问题

| 编号 | 问题 | 影响 | 处理建议 |
| --- | --- | --- | --- |
| R-001 | JMeter 未保存业务响应体 | 无法直接从 JTL 判断业务 code 和 message | 增加 JSON Extractor 或 sample variable |
| R-002 | 原始 JTL 含 subresult | 若直接按行统计会导致请求数翻倍 | 统计时只采用 13 列主样本 |
| R-003 | 售空后请求数量较大 | 全局 P99 会被售空后拒绝请求污染 | 性能评价区分售空前和售空后 |
| R-004 | `203B` 业务拒绝未定位到具体业务码 | 影响真实用户成功率解释 | 保存业务响应体并分析拒绝原因 |

### 7.2 剩余风险

| 风险 | 说明 | 后续措施 |
| --- | --- | --- |

## 8 测试结论

本轮测试表明，万花筒系统已具备可重复构建、基础单元测试验证和核心下单压力测试结果追溯能力。后端 Java 17 构建路径、前端 Vite 构建、前端镜像构建和 Compose 配置检查均已通过；公共契约、业务枚举、字符串工具和自动选座算法已有自动化测试覆盖。

核心下单 V4 接口在 500 并发线程抢 1500 座的压测中，售空前真实成交请求数为 1500，未出现超卖。售空前 P99 为 2.249s，峰值 1s 成交吞吐为 268 orders/s，说明当前分片、Redis Lua 和异步建单设计能够支撑本轮高并发一致性验证。

当前测试工作的主要不足是集成测试、故障注入测试和系统资源观测尚未完全补齐。后续应继续完善端到端自动化测试、业务响应字段采集、故障恢复验证和生产级容量评估。
