## 关键发现

### 1. 零超卖已验证
- `total_remain` 从 1500 严格递减至 0
- DECRBY 原子操作确保无竞态条件
- 3 实例首次同时报 "余票数量不足" 在 16:31:24，时间戳一致

### 2. 快速拒绝生效
- 售罄后 142,424 次请求被 O(1) GET 拦截在 Lua 之前
- 最后 10s P50=69ms 证明极快路径没问题

### 3. 长尾根因：Redis Lua 排队
- 整个稳态期间 P99 持续在 7-8s，并非冷启动
- 500 并发 × 10 shard = 每 shard ~50 并发竞争 Lua 脚本
- Redis 单线程串行执行 hincrby + hvals + hdel + hmset
- 售罄后 P99 反而更高（7,179ms vs 3,975ms），因为 500 线程
  全部在忙等 total_remain 读数，程序实例本身被请求压满

### 4. 发现预存 Bug：Lua cjson 大数精度丢失
- Snowflake ID (18 位) 经 Lua `cjson.decode()` 后丢失低 6 位精度
  `2181859569805017088` → `2.181859569805e+18` (截断)
- 导致 `hdel`/`hmset` 计算的目标 key 与预期不符
- 座位在 Redis NO_SOLD→LOCK 移动失败
- 此 bug 与本次改造无关，是系统原有问题

## 结论

- 分片+Lua+快速拒绝架构在 500 并发下可工作
- 稳态 TPS ~191/s (JMeter 汇总器报告)
- 无超卖，快速拒绝有效
- P99 7.2s 偏高，主因是单 Redis 单线程执行 Lua 造成的排队
- 改善方向：增加 Redis 实例数、引入多 Redis 分片、或减少 Lua 脚本粒度
