# 万花筒 Dismai
## 高并发在线票务系统演示与分布式可靠性答辩

分布式软件原理与技术 · Demo Ready

---

# 演示目标

- 展示在线票务系统的完整购票链路：浏览、选座、下单、支付、订单查询
- 解释分布式组件如何服务高并发抢票：Gateway、Redis、Kafka、MySQL、ES
- 回答答辩高频问题：防刷、锁座、Kafka 故障、Redis/数据库故障

---

# 系统架构

Vue3 前端 -> Nginx/Gateway -> User / Program / Order / Pay

- Gateway：统一入口、路由、限流
- Program：节目、票档、座位、库存缓存
- Order：订单落库、支付状态、取消补偿
- Redis/Redisson：座位状态、库存、幂等、分布式锁、延迟队列
- Kafka：异步建单、削峰填谷
- MySQL + ShardingSphere：最终事实源，分库分表

---

# 核心下单链路

1. 参数校验与业务校验
2. `@RepeatExecuteLimit` 防重复提交
3. 下单 V4 随机选择库存分片
4. 本地分片锁降低 JVM 内并发冲突
5. Redis Lua 原子完成余票扣减和锁座
6. Kafka 异步创建订单
7. 延迟队列释放超时未支付座位

---

# 如何防止黑产刷流量

- 入口层：Gateway 接入 Sentinel，可按路由/API 配置 QPS 规则
- 用户层：验证码、Redis 计数、注册频率控制
- 业务层：请求签名开关、重复执行限制、管理员权限隔离
- 缓存层：节目详情布隆过滤器拦截不存在 ID，减少缓存穿透

答辩口径：不是靠单个组件防刷，而是入口、身份、行为、业务资源四层防护。

---

# 如何实现锁座

锁座是一个状态机：

`未售 -> 锁定 -> 已售 / 释放`

- Redis Hash 保存未售座位、锁定座位和余票数
- Lua 脚本一次完成：校验余票、扣减库存、迁移座位状态
- 下单失败或 Kafka 发送失败：回滚锁座和库存
- 未支付超时：延迟队列触发取消，释放座位

---

# Kafka 挂了怎么办

- 发送失败：失败回调立即回滚 Redis 中的锁座和库存
- 消费失败：抛出异常交给 Kafka/Spring 消费重试
- 消息过期：超过阈值后丢弃，并回滚节目侧锁座数据
- 长时间不可用：下单失败，保护一致性，宁可少卖不可超卖

生产增强：Outbox、本地消息表、死信队列、补偿任务。

---

# Redis 挂了怎么办

- 写链路强依赖 Redis，因为锁座和扣库存必须原子化
- Redis 不可用时不绕过缓存直接卖票，应该熔断下单
- 读链路可用本地缓存或旧缓存降级展示
- Redis 恢复后以数据库为事实源重建节目、票档、座位缓存

答辩口径：Redis 故障时优先保护不超卖，而不是强行提高可用性。

---

# 数据库挂了怎么办

- 短故障：Kafka 缓冲订单消息，数据库恢复后继续消费
- 消费异常：订单消费者抛异常，消息重试
- 长故障：写链路熔断，避免大量锁座长时间堆积
- 恢复后：以订单表为准，通过对账/初始化任务校正 Redis

数据库是最终事实源，缓存和消息队列服务的是吞吐与削峰。

---

# 一致性模型

- 强约束：同一座位不能重复锁定，余票不能扣成负数
- 最终一致：订单落库、支付状态、超时取消可以异步完成
- 补偿机制：Kafka 失败回滚、过期消息释放、支付/取消加分布式锁

一句话：先在 Redis 确定座位抢占权，再让订单系统追上。

# 答辩速答

- 防刷：Gateway Sentinel + 验证码/计数 + 幂等限制 + 布隆过滤器
- 锁座：Redis Lua 原子扣库存并迁移座位状态
- Kafka 挂：失败回滚，消费重试，长期不可用则下单失败
- Redis 挂：写链路熔断，不绕过锁座状态机
- DB 挂：Kafka 短暂缓冲，长故障熔断，恢复后对账校正

---

# 结论

这个系统的重点不是堆组件，而是在高并发和故障场景下守住三条底线：

- 不超卖
- 不重复下单
- 故障可解释、可回滚、可恢复
