Problem

票务系统为什么适合讨论分布式

抢票不是普通 CRUD，它把流量、状态、一致性和故障处理集中到同一条链路上。

业务约束

热点极端集中

  • 热门节目开售瞬间 QPS 激增
  • 同一票档和座位被大量请求争抢

库存不可超卖

  • 一个座位只能属于一个订单
  • 余票不能出现负数

用户体验敏感

  • 页面要快速展示
  • 下单不能长期阻塞

依赖不可完全可靠

  • Kafka 可能延迟或不可用
  • Redis/DB 可能故障
本项目的核心问题：如何在高并发和局部故障下，仍然守住座位状态的一致性。

07 / 20
Seat Lock

如何实现锁座

锁座不是简单加锁，而是座位状态机的原子迁移。

状态机

未售

Redis: PROGRAM_SEAT_NO_SOLD_HASH

用户可选择，余票可扣减

锁定

Lua 原子执行：校验余票 + 扣库存 + 迁移座位

订单待支付，超时前别人不可再买

已售 / 释放

支付成功转已售；失败、取消、超时则释放回未售

通过延迟队列和回滚逻辑补偿

1. 查询座位从 Redis 拉取未售/锁定状态
2. 用户提交携带节目、票档、座位、购票人
3. Lua 锁座原子校验并扣减余票
4. Kafka 建单异步写 Order 服务
5. 延迟取消未支付释放座位
6. 支付确认订单状态变更并最终一致
08 / 20
Correctness

为什么这个锁座方案不会超卖

把“互斥”问题转化为 Redis 单线程脚本中的状态迁移问题。

正确性说明

并发风险

  • 两个用户同时选择同一座位
  • 多个用户同时购买同一票档余票
  • 订单服务写库慢于用户请求速度
  • Kafka 或数据库短暂故障导致状态不一致

系统约束

  • 座位是否可卖以 Redis 状态为准
  • 扣库存和锁座在同一 Lua 脚本内完成
  • 失败时只做补偿释放，不凭空创建订单
  • 订单异步落库，但座位抢占权先被确定
核心不变量：任意时刻，一个座位只能存在于“未售、锁定、已售”中的一个状态集合。

Redis Failure

Redis 挂了怎么办

Redis 是购票写链路的状态机载体。故障时继续卖票会破坏一致性。

缓存故障

不能绕过 Redis

  • 余票状态在 Redis
  • 座位状态迁移依赖 Lua 原子性
  • 绕过后数据库无法承受抢票峰值

故障时策略

  • 写链路熔断，下单失败
  • 提示用户稍后重试
  • 避免超卖和悬挂状态

恢复后策略

  • 以数据库为事实源
  • 重建节目、票档、座位缓存
  • 必要时执行对账修复
关键回答

Redis 故障时，读链路可以展示降级内容；写链路必须停止抢票，因为锁座状态机不可用。

11 / 20
Database Failure

数据库挂了怎么办

数据库是最终事实源；故障处理重点是限制写入窗口并保证恢复后可校正。

事实源故障

短暂故障

  • Kafka 消息可短时间缓冲
  • 订单消费者失败后重试
  • 数据库恢复后继续落库

长时间故障

  • 写链路应熔断
  • 防止锁座无限堆积
  • 对用户明确返回失败

恢复与校正

  • 以订单表为准
  • 校正 Redis 座位和余票
  • 重新执行缓存初始化任务
数据库故障的核心原则：短故障靠消息缓冲，长故障靠熔断止损，恢复后靠对账修正。
12 / 20
Consistency

一致性模型与补偿机制

本系统不是所有步骤都强一致，而是对座位抢占强约束，对订单落库最终一致。

CAP 取舍

强约束部分

  • 同一座位不能重复锁定
  • 同一票档余票不能扣成负数
  • 同一用户/节目重复请求被限制

最终一致部分

  • 订单异步落库
  • 支付状态异步更新
  • 超时取消异步释放

补偿动作

  • Kafka 失败回滚缓存
  • 消息过期释放座位
  • 支付/取消加订单号维度锁
一句话

先在 Redis 确定座位抢占权，再让订单系统通过 Kafka 和补偿机制追上。
