# 万花筒 Dismai 演示 PPT 讲稿

## 1. 开场
本项目是高并发在线票务系统，目标是把分布式课程里的网关、缓存、消息队列、分布式锁、分库分表、限流降级放到一个完整购票链路里。

## 2. 系统架构
前端通过 Nginx/Gateway 进入后端微服务。节目读链路走 Program 服务、缓存和 ES；下单写链路先在 Redis 里锁座和扣库存，再通过 Kafka 异步创建订单，最终写入 MySQL 分库分表。

## 3. 防止黑产刷流量
我们的防刷不是单点能力，而是分层防护：Gateway 集成 Sentinel，在入口限流；注册/登录有验证码和 Redis 计数；业务接口用 `@RepeatExecuteLimit` 防重复提交；节目详情用布隆过滤器拦截不存在 ID，避免穿透缓存和数据库。

## 4. 锁座实现
锁座是一个状态机：未售、锁定、已售/释放。系统用 Redis Hash 保存不同状态的座位集合，用 Lua 脚本把“校验余票、扣库存、未售座位转锁定座位”放在一次原子执行里。这样即使并发很高，也不会出现两个请求同时拿到同一座位。

## 5. Kafka 挂了怎么办
如果 Kafka 发送失败，Program 服务会执行失败回调，回滚刚才在 Redis 里的锁座和库存扣减，然后返回下单失败。Kafka 消费失败时抛出异常交给消费端重试；如果消息延迟超过阈值，订单消息会被判定过期，并回滚节目侧座位数据。答辩时强调：我们优先保证不超卖，Kafka 完全不可用时宁可失败，不让缓存进入错误状态。

## 6. Redis 挂了怎么办
购票写链路强依赖 Redis，因为座位锁定和库存扣减都在 Redis 原子脚本里完成。Redis 不可用时，正确策略不是绕过 Redis 直接写库，而是熔断下单，避免超卖。读链路可以用本地缓存或旧缓存降级展示；恢复后通过数据库和初始化任务重建缓存。

## 7. 数据库挂了怎么办
数据库是最终事实源。短时间数据库故障时，Kafka 可以缓冲部分订单消息，恢复后继续消费；消费异常会重试。长时间故障时，应该熔断写链路，避免锁座堆积。恢复后以订单表为准，通过对账或初始化任务校正 Redis 缓存。
