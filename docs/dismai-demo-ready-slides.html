Kafka 挂了怎么办

当前策略：失败即回滚，宁可少卖，不可超卖

07

发送失败

  • 失败回调记录错误
  • 回滚 Redis 锁座和库存
  • 返回下单失败，用户可重试

消费异常

  • 抛出 RuntimeException
  • 交给消费端重试
  • 过期消息取消节目侧数据

长期不可用

  • 下单链路失败
  • 保护座位一致性
  • 生产增强 Outbox/死信队列
核心回答：Kafka 不可用时不继续占座，系统优先保证不会超卖。

Redis 挂了怎么办

购票写链路强依赖 Redis，故障时必须降级为不可下单

08

为什么不能绕过

  • 座位状态在 Redis
  • 库存扣减依赖 Lua 原子性
  • 绕过会增加超卖风险

已有措施

  • Compose 健康检查
  • 服务依赖 Redis healthy
  • 失败返回业务异常

恢复策略

  • 读链路旧缓存降级
  • 写链路熔断
  • 恢复后从 DB 重建缓存
答辩口径：Redis 故障时优先保护不超卖，而不是强行提高写入可用性。

数据库挂了怎么办

数据库是最终事实源；写入失败必须让缓存回到可售状态

09

短故障

  • Kafka 暂存订单消息
  • 数据库恢复后继续消费
  • 消费异常触发重试

长故障

  • 写链路熔断
  • 避免锁座堆积
  • 提示用户稍后重试

恢复后

  • 以订单表为准
  • 对账校正 Redis
  • 重建节目和座位缓存
数据库负责最终事实，Redis 和 Kafka 负责高峰期的吞吐与削峰。

一致性模型

系统选择最终一致性，但对“不能超卖”保持强约束

10

强约束

  • 同一座位不能重复锁定
  • 余票不能扣成负数
  • 重复提交会被拦截

最终一致

  • 订单异步落库
  • 支付状态异步更新
  • 超时取消异步释放

补偿机制

  • Kafka 失败回滚
  • 过期消息释放
  • 支付/取消加分布式锁
一句话：先在 Redis 确定座位抢占权，再让订单系统追上。

答辩 Q&A 速答

把复杂问题压缩成可讲清楚的三句话

13
如何防刷？Gateway Sentinel 限流、验证码/计数、请求签名、幂等限制、布隆过滤器，多层挡流量。
如何锁座？Redis Lua 原子扣库存并迁移座位状态，失败和超时再补偿释放。
Kafka 挂？发送失败回滚，消费失败重试，长期不可用则下单失败。
Redis 挂？写链路熔断，不绕过锁座状态机；读链路可旧缓存降级。
DB 挂？短故障 Kafka 缓冲，长故障熔断，恢复后以订单表对账校正。
一致性？座位抢占是强约束，订单落库和支付状态走最终一致。

结论

演示重点不是组件堆砌，而是故障下如何保护业务不出错
