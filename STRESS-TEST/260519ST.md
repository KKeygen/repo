## 二、指标口径

| 指标 | 含义 | 记录方式 |
| --- | --- | --- |
| QPS | HTTP 层每秒请求数 | JMeter Summary Report / HTML Report |
| TPS | 业务成功事务数每秒 | `code=0` 的创建订单请求数 / 秒 |
| 平均响应时间 | HTTP 请求平均耗时 | JMeter `Average` |
| P90 | 90% 请求响应时间 | JMeter percentile |
| P95 | 95% 请求响应时间 | JMeter percentile |
| P99 | 99% 请求响应时间 | JMeter percentile |
| 错误率 | HTTP 失败或业务失败比例 | HTTP 非 2xx、`code != 0`、超时 |
| 成功订单数 | 实际成功落库订单数 | MySQL 订单表统计 |
| 超卖情况 | 成功订单数是否超过库存 | 对比票档库存和成功订单数 |
| MQ 积压 | Kafka 异步消费是否滞后 | 请求返回订单号与订单落库数量差异 |

### 失败原因补充

单条请求：

```bash
curl -i -X POST 'http://127.0.0.1:6085/Dismai/program/program/order/create/v4' \
  -H 'Content-Type: application/json' \
  -H 'no_verify: true' \
  --data '{"programId":32,"userId":3000000000000000001,"ticketUserIdList":[3100000000000000001],"ticketCategoryId":35,"ticketCount":1}'
```

返回：

```http
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8

{"code":"-100","message":"系统错误，请稍后重试!","data":null}
```

`program-service` 日志核心异常：

```text
ERR user_script:1: Bloom filter config has been changed
Dismai-program-detail-bloom-filter
```

原因判断：

压测数据扩容后执行了 Redis `FLUSHALL`，清除了运行中服务依赖的布隆过滤器配置与缓存数据。`program-service` 中 Redisson BloomFilter 对象仍按服务启动时的状态访问 Redis，导致布隆过滤器配置校验失败。

处理方式：

重启 `program-service`，让 `ProgramBloomFilterInit` 重新初始化节目详情布隆过滤器后，再重新执行单条 curl 和 JMeter 压测。

### 恢复后单条验证

重启 `program-service` 后，重新执行单条创建订单 V4 请求：

```bash
curl -i -X POST 'http://127.0.0.1:6085/Dismai/program/program/order/create/v4' \

```

```http
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8

{"code":"0","message":"","data":"2056442955505340425"}
```

当前结论：

`program-service` 重启后，节目详情布隆过滤器恢复正常，单条创建订单 V4 请求成功。
