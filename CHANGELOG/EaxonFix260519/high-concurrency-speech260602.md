## 开场

各位老师、同学大家好，我这次主要汇报我们平台中订单、支付以及高并发支撑框架这一块的设计和实现。

这个项目的核心业务场景是秒杀类购票或电商下单。它和普通商品下单最大的区别是：请求量会在某个时间点瞬间冲高，而且库存、座位、购票人、订单、支付状态之间必须保持相对一致。如果我们只用传统的“查数据库库存、扣数据库库存、生成订单”这一套，在高并发下很容易遇到超卖、重复下单、数据库压力过大、支付回调状态错乱等问题。

所以我们这一块的设计目标可以概括为三句话：

1. 高并发请求先进 Redis，用 Lua 保证核心扣减动作原子化。
2. 订单和支付服务用分布式锁、重复执行限制、延迟队列和消息队列保证链路推进。
3. 支付成功、取消超时、退款等后续动作反向修正缓存和数据库，尽量保证最终一致。

## 一、我负责的模块边界

我负责的后端范围主要分成四类。

第一类是订单服务：

```text
repo/dismai-server/dismai-order-service
```

它负责订单落库、订单状态流转、订单取消、支付前校验、支付回调处理、订单详情查询，以及订单状态变化后对节目库存缓存的反向恢复。

第二类是支付服务：

```text
repo/dismai-server/dismai-pay-service
```

它负责生成支付单、调用支付宝 SDK、处理支付回调验签、主动查询交易状态、退款以及支付账单查询。

第三类是 Redis 工具框架：

```text
repo/dismai-redis-tool-framework
```

它封装了项目中常用的 Redis 操作，包括普通 key、hash、list、set、zset、Lua 脚本执行，以及 Redis Stream 的推送和消费封装。

第四类是 Redisson 高并发支撑框架：

```text
repo/dismai-redisson-framework
```

它提供了分布式锁、重复执行限制、本地锁缓存、延迟队列、布隆过滤器等能力。订单和秒杀链路里面大量用到了这些能力。

## 二、核心技术栈

这一部分的技术栈主要包括：

- Java 17：后端服务运行基础。
- Spring Boot 3.3：微服务应用框架。
- Spring Cloud Alibaba Nacos：服务注册与发现。
- OpenFeign：微服务之间的 RPC 调用，例如订单服务调用支付服务。
- MyBatis-Plus：业务表的 ORM 操作。
- ShardingSphere-JDBC：订单表、支付表的分库分表。
- Redis：高并发缓存、库存、座位状态、重复提交标记。
- Redis Lua：库存扣减、座位迁移、身份证占用等核心动作的原子执行。
- Redisson：分布式锁、延迟队列、本地锁和 Redis 协同能力。
- Kafka：异步创建订单，削峰解耦。
- 支付宝 SDK：第三方支付、回调验签、主动查单和退款。
- UidGenerator：生成订单号、主键 id，配合分库分表路由。

如果只从秒杀链路来看，最关键的不是某一个单独技术，而是 Redis Lua、Redisson 锁、Kafka、延迟队列、订单状态机这几部分如何配合。

## 三、秒杀下单主链路

我先讲最核心的秒杀下单链路。

用户在前端提交购票请求后，首先进入 Program 服务的下单策略。虽然我主要负责订单和支付域，但真正的秒杀库存前置处理发生在 Program 服务中，订单服务承接它处理后的结果。

目前 Program 服务提供了多个下单版本：

- V1：按节目维度加分布式锁，粒度比较粗。
- V2：按节目和票档维度做本地锁加分布式锁，降低锁冲突。
- V3：本地锁加 Redis Lua 原子处理库存和座位。
- V4：本地锁加 Redis Lua，再通过 Kafka 异步创建订单。

可以把 V4 理解为当前更接近高并发秒杀的方案。

### 1. Redis Lua 原子扣减库存

Program 服务会调用：

```text
dismai-server/dismai-program-service/src/main/resources/lua/programDataCreateOrderResolution.lua
```

这个 Lua 脚本做了几件关键事情。

第一，它判断身份证是否重复购买。脚本会读取：

```text
PROGRAM_ID_CARD:{programId}
```

如果本次请求里的身份证在同一个节目下已经存在，就直接返回重复购票错误。

第二，它检查余票是否充足。余票存在 Redis hash 中，key 由：

```text
PROGRAM_TICKET_REMAIN_NUMBER_HASH_RESOLUTION
```

构建出来。脚本会先判断本次购买数量是否大于 Redis 中的剩余数量。

第三，如果是选座购票，它会检查用户指定的座位是否仍在未售集合中。如果座位不存在，说明已经被别人锁定或售出。

第四，如果是不选座购票，它会在未售座位集合中自动寻找相邻座位。

第五，真正扣减时，脚本会在 Redis 里一次性完成：

- 余票数量 `HINCRBY -count`
- 从未售座位 hash 删除座位
- 把座位写入锁定座位 hash
- 把身份证写入节目身份证占用 hash
- 返回本次成功锁定的座位列表

这里的关键点是：这些动作在一个 Lua 脚本里执行，Redis 会把它当成一个原子操作。也就是说，在脚本执行期间，不会有另一个请求插进来抢同一批座位。这就是我们防止超卖和重复锁座的第一道核心保障。

### 2. 构建订单参数

Lua 成功后，Program 服务拿到锁定成功的座位列表，然后构建 `OrderCreateDto`。

这里会生成订单号、节目 id、用户 id、节目标题、演出时间、订单金额，以及每一个购票人的票务信息。

订单号通过 UidGenerator 生成，它不只是一个随机数，还要兼顾后续 ShardingSphere 的分库分表路由。

### 3. 同步建单与异步建单

同步建单会直接通过 OpenFeign 调用订单服务：

```text
orderClient.create(orderCreateDto)
```

异步建单则通过 Kafka 发送创建订单消息：

```text
CreateOrderSend -> Kafka -> CreateOrderConsumer -> OrderService.createMq
```

异步建单的好处是削峰。前端请求只要完成 Redis 原子锁座和 Kafka 投递，就可以先返回订单号。真正的订单落库由订单服务消费者慢慢处理。

目前订单服务的消费者 `CreateOrderConsumer` 已经使用 `Acknowledgment` 手动确认。它会在处理完成后再 ack。这样比自动提交 offset 更安全，因为如果订单落库失败，消息不会轻易被确认掉。

## 四、订单服务核心链路

订单服务入口主要在：

```text
dismai-server/dismai-order-service/src/main/java/com/dismai/service/OrderService.java
```

### 1. 创建订单

订单服务的 `create()` 方法会做三件事：

第一，根据订单号查询是否已存在订单。这个是数据库层面的幂等保护。

第二，插入主订单表 `d_order`，再批量保存购票人订单表 `d_order_ticket_user`。

第三，给用户在某个节目下的订单数量计数：

```text
ACCOUNT_ORDER_COUNT:{userId}:{programId}
```

这个计数后面会用于限制用户对同一节目购买数量。

如果是异步建单，`createMq()` 在创建订单后还会写一个短期 Redis 标记：

```text
ORDER_MQ:{orderNumber}
```

前端或其他服务可以用这个标记判断异步订单是否已经被消费者处理。

### 2. 延迟取消订单

用户锁座成功后，如果一直不支付，系统不能让座位永久锁住。所以 Program 服务在建单后会发送延迟取消消息，订单服务消费后调用：

```text
OrderService.cancel()
```

这里用了两个注解：

```java
@RepeatExecuteLimit
@ServiceLock
```

`@RepeatExecuteLimit` 用于防止同一个订单号的取消逻辑被重复触发。  
`@ServiceLock` 用于按订单号加分布式锁，保证支付回调、主动取消、延迟取消不会同时改同一个订单。

### 3. 订单状态更新与缓存反向恢复

订单状态变化时，核心方法是：

```text
updateOrderRelatedData(orderNumber, orderStatus)
```

它先更新订单主表，再更新购票人订单表，然后根据订单状态去修正 Redis 中的座位和余票。

如果订单取消：

- 订单状态改为已取消。
- 购票人订单状态也改为已取消。
- 用户节目订单计数减掉本次票数。
- Redis 中的座位从“锁定集合”移回“未售集合”。
- 余票数量加回去。
- 身份证占用释放。

如果订单支付成功：

- 订单状态改为已支付。
- 购票人订单状态改为已支付。
- Redis 中的座位从“锁定集合”移到“已售集合”。
- 再发送延迟消息，推动 Program 服务侧的数据最终落库或后续处理。

这里用到的订单侧 Lua 是：

```text
dismai-server/dismai-order-service/src/main/resources/lua/OrderProgramDataResolution.lua
```

这个脚本会根据订单状态做反向迁移。取消时它会释放座位、加回余票、删除身份证占用；支付成功时它会把锁定座位移入已售集合。

## 五、支付链路

支付链路分成发起支付、支付回调、主动查单、退款四个环节。

### 1. 发起支付

前端点支付后，请求先到订单服务：

```text
OrderService.pay()
```

订单服务会按订单号加分布式锁，检查订单是否存在，是否已取消，是否已支付，是否已退款，以及前端传入金额是否和订单金额一致。

检查通过后，订单服务通过 Feign 调用支付服务：

```text
PayClient.commonPay(payDto)
```

支付服务的 `PayService.commonPay()` 也会按订单号加锁，避免同一订单重复创建支付单。然后它通过策略模式找到支付渠道：

```text
PayStrategyContext -> AlipayStrategyHandler
```

目前主要实现的是支付宝网页支付。支付服务创建本地支付账单 `d_pay_bill`，状态是未支付，然后调用支付宝 SDK 返回支付页面 body。

### 2. 支付宝回调

支付宝支付完成后，会回调订单服务：

```text
OrderService.alipayNotify()
```

这里仍然按订单号加锁。这样可以避免支付回调、主动查单、延迟取消同时改一个订单。

如果订单已经被延迟取消，订单服务会调用支付服务退款。  
如果订单还未支付，订单服务会把支付宝参数交给支付服务：

```text
PayService.notify()
```

支付服务会做验签、金额校验、sellerId 校验、appId 校验、交易状态校验。校验通过后，将支付账单状态改为已支付。

订单服务拿到支付服务返回成功后，再把订单状态改为已支付，并把 Redis 中的座位从锁定集合移动到已售集合。

### 3. 主动查单

除了异步回调，我们还提供主动查单：

```text
OrderService.payCheck()
```

它会调用支付服务：

```text
PayService.tradeCheck()
```

支付服务再调用支付宝交易查询接口。如果支付宝侧已经支付，但本地支付账单状态还没有更新，就会更新本地支付状态。订单服务再根据支付状态更新订单状态。

这相当于对支付回调的一种补偿机制。因为第三方回调可能延迟，也可能因为网络原因失败，所以主动查单可以补齐支付状态。

### 4. 退款

退款主要发生在这种场景：订单已经因为超时取消，但支付宝支付回调后来才到。这个时候不能把订单重新变成已支付，而是要对这笔支付做退款。

订单服务检测到订单已取消后，会调用：

```text
PayService.refund()
```

支付服务校验账单存在、账单是已支付状态、退款金额不大于支付金额，然后通过支付宝 SDK 发起退款，并更新本地退款账单。

## 六、高并发支撑框架怎么工作

### 1. Redis 工具框架

Redis 工具框架主要封装在：

```text
dismai-redis-tool-framework
```

它做了几类事情：

- 统一 Redis key 构建。
- 封装 String、Hash、List、Set、ZSet 的常用操作。
- 支持 JSON 序列化和反序列化。
- 暴露 `getInstance()` 给 Lua 脚本执行场景使用。
- 封装 Redis Stream 的生产和消费。

统一 key 构建很重要。项目中 Redis key 会带系统前缀，例如 `Dismai-xxx`。这样本地、测试、生产，或者不同业务隔离时，可以避免 key 冲突。

### 2. Redisson 分布式锁框架

Redisson 锁框架主要在：

```text
dismai-redisson-framework/dismai-redisson-service-framework/dismai-service-lock-framework
```

核心注解是：

```java
@ServiceLock
```

它通过 AOP 拦截方法调用，根据注解里的业务名和 SpEL 参数生成锁 key，然后调用 Redisson 加锁。

比如订单支付、订单取消、支付检查都会按订单号加锁。这种锁粒度比较合理：不同订单之间可以并发，同一个订单的状态更新串行。

锁 key 的生成支持 SpEL，例如：

```java
keys = {"#orderPayDto.orderNumber"}
```

这意味着我们不用手写大量重复的加锁代码，业务方法只要声明自己的锁维度即可。

### 3. 重复执行限制

重复执行限制在：

```text
dismai-redisson-framework/dismai-redisson-service-framework/dismai-repeat-execute-limit-framework
```

核心注解是：

```java
@RepeatExecuteLimit
```

它的作用和普通分布式锁不同。分布式锁解决的是“同一时间只能有一个线程执行”。重复执行限制解决的是“这件事已经做过了，短时间内不要再做”。

比如取消订单，如果延迟队列触发了一次，用户手动取消又触发一次，那么同一个订单号的取消流程不应该重复推进。这个时候 `@RepeatExecuteLimit` 会用本地锁加 Redisson 标记来挡住重复执行。

### 4. 本地锁加分布式锁

在秒杀场景中，如果每个请求都直接争抢 Redisson 分布式锁，Redis 自身也会承受很大压力。

所以部分下单策略会先使用本地锁：

```text
LocalLockCache -> ReentrantLock
```

同一台机器上的请求先在 JVM 内部排队，减少跨网络的分布式锁竞争。多台机器之间再依赖 Redisson 或 Redis Lua 保证全局一致。

这种设计的思路是：本地能挡住的并发先在本地挡，全局必须一致的地方再交给 Redis。

### 5. 延迟队列

延迟队列基于 Redisson delayed queue 封装：

```text
dismai-redisson-framework/dismai-service-delay-queue-framework
```

订单链路里主要有两个延迟用途：

第一是订单超时取消。用户下单锁座后，如果超过支付时间未支付，就要自动取消并释放库存。

第二是支付成功后延迟同步节目数据。订单支付成功后，Redis 里的座位已经从锁定改成已售，但数据库或节目侧后续数据也需要推进，所以用延迟消息做异步处理。

## 七、我们如何处理数据一致性

我这里用一个口头问题来讲。

**问题一：Redis 扣库存成功了，但订单没创建成功怎么办？**

我们的处理思路是补偿。Program 服务在 Redis 中锁座和扣余票后，如果同步 RPC 创建订单失败，会立即调用缓存反向恢复逻辑，把座位从锁定集合移回未售集合，把余票加回去。

如果是异步 Kafka 建单，订单服务消费者会判断消息延迟。如果消息超过可接受时间，就不再创建订单，而是调用缓存反向恢复逻辑释放座位和身份证占用。

**问题二：订单已经取消了，但支付回调又来了怎么办？**

订单服务的支付回调入口会先按订单号加锁，然后查订单状态。如果订单已经取消，不会把订单改回已支付，而是调用支付服务退款。这样可以避免“超时取消”和“支付回调延迟”之间的状态冲突。

**问题三：为什么不是直接扣数据库库存？**

因为秒杀的并发峰值很高，数据库行锁和事务会成为瓶颈。我们的设计是把最热的库存判断、座位锁定、余票扣减放到 Redis Lua 里完成。Redis 单线程执行 Lua，可以用很低成本保证这段逻辑的原子性。数据库主要承接最终订单落库和状态记录。

**问题四：为什么还需要分布式锁，Lua 不是已经原子了吗？**

Lua 解决的是 Redis 里库存和座位迁移的原子性。  
分布式锁解决的是订单状态、支付状态这种跨服务、跨数据库、跨第三方支付的状态流转问题。

比如同一个订单，可能同时发生用户取消、延迟取消、支付回调、主动查单。如果没有按订单号加锁，这几个动作可能同时改订单状态。Lua 管不到数据库状态流转，所以这里还需要 Redisson 锁。

**问题五：为什么需要 Kafka 异步建单？**

同步建单会把前端请求一直阻塞到订单落库完成。高峰时数据库压力会直接影响用户体验。异步建单把 Redis 锁座和订单落库拆开：前端先拿到订单号，订单服务消费者慢慢落库。这样可以削峰，但代价是必须处理消息可靠性、订单查询状态和失败补偿。

## 八、秒杀链路可以用这张图理解

```text
用户请求
  |
  v
Program 下单策略
  |
  v
Redis Lua 原子处理
  |-- 校验余票
  |-- 校验座位状态
  |-- 校验身份证重复购买
  |-- 扣减余票
  |-- 未售座位 -> 锁定座位
  |-- 写入身份证占用
  |
  v
构建 OrderCreateDto
  |
  +--> 同步 RPC 建单 -> OrderService.create
  |
  +--> Kafka 异步建单 -> CreateOrderConsumer -> OrderService.createMq
  |
  v
发送延迟取消消息
  |
  v
用户支付
  |
  v
OrderService.pay -> PayService.commonPay -> 支付宝
  |
  v
支付宝回调 / 主动查单
  |
  v
订单状态更新
  |
  +--> 支付成功：锁定座位 -> 已售座位
  |
  +--> 超时取消：锁定座位 -> 未售座位，余票加回，身份证释放
```

## 九、当前链路的技术难点

第一，库存不能超卖。  
这里靠 Redis Lua 保证扣减、座位迁移、身份证占用写入在一个原子脚本里完成。

第二，同一个订单状态不能乱跳。  
这里靠订单号维度的 `@ServiceLock`，把取消、支付、回调、查单这些状态更新串行化。

第三，用户不能重复提交。  
这里靠 `@RepeatExecuteLimit` 和订单号唯一校验。对于下单，还会按用户和节目维度做重复限制。

第四，异步建单必须补偿。  
Kafka 帮我们削峰，但消息失败、消息延迟、订单落库失败都要考虑。当前消费者已经使用手动 ack，处理完成再确认，避免自动提交造成丢单。

第五，第三方支付不是强实时的。  
支付宝回调可能延迟，主动查单也可能和回调同时发生。因此订单和支付都必须具备幂等判断，不能简单地“收到成功就直接改状态”。

第六，缓存和数据库最终一致。  
秒杀瞬间优先相信 Redis，订单落库和节目数据更新异步推进。订单状态变化时，再通过 Lua 把 Redis 中的座位和余票修正到正确状态。

## 十、当前实现的阶段性总结

从完成度上看，订单、支付、高并发支撑框架已经形成了比较完整的主链路：

- 秒杀库存和座位处理已经从数据库前置到 Redis。
- 核心扣减逻辑使用 Lua 保证原子性。
- 订单服务能创建订单、取消订单、支付、查单、处理支付宝回调。
- 支付服务能创建支付单、调用支付宝、验签回调、查单、退款。
- Redisson 框架提供了分布式锁、重复执行限制和延迟队列。
- Kafka 异步建单已经接入，并且当前消费者采用手动确认。

但从工程成熟度上看，仍然有需要继续加强的地方：

- 支付账单状态流转还可以继续增强幂等条件。
- Redis Stream 和延迟队列异常恢复可以继续完善。
- 金额字段、敏感配置、测试覆盖需要系统性补齐。
- 秒杀链路还需要压测，用数据验证当前锁粒度、Lua 执行耗时和 Kafka 消费能力。

## 结尾

总结一下，我们这一块不是单纯做一个订单接口或者支付接口，而是在解决一个高并发秒杀场景下的完整交易链路问题。

这个链路的核心思想是：热点请求先进入 Redis，关键扣减用 Lua 保证原子性，订单状态用分布式锁串行化，慢操作通过 Kafka 和延迟队列异步推进，支付状态通过回调和主动查单双通道兜底，最终让 Redis、订单库、支付库和节目数据达到一致。

这就是我负责的订单、支付和高并发支撑框架部分的核心设计与实现。谢谢大家。
