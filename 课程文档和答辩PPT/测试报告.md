## 1 测试概述

本报告记录万花筒高并发在线票务服务系统在课程大作业提交前完成的测试工作。测试对象包括后端 Maven 多模块工程、Vue 3 前端工程、核心业务单元测试、集成测试入口、故障注入脚本和 JMeter 高并发下单压测方案。

测试目标如下：

| 编号 | 测试目标 | 说明 |
| --- | --- | --- |
| T-001 | 验证源码完整性 | 后端、前端、SQL、测试代码结构完整，能够支撑课程评审 |
| T-002 | 验证核心逻辑正确性 | 覆盖统一响应、业务码、订单版本、字符串工具、自动选座等基础逻辑 |
| T-003 | 验证业务链路可测性 | 提供用户登录、节目查询、锁座下单、订单取消、支付回调等集成测试入口 |
| T-004 | 验证高并发一致性 | 使用 JMeter 对下单 V4 接口进行并发压测，重点检查是否超卖 |
| T-005 | 验证故障场景 | 记录并验证 Kafka、Redis、MySQL 等中间件异常场景 |

### 5.2 高并发下单压测结果

JMeter 压测场景针对 `POST /Dismai/program/program/order/create/v4` 下单接口，使用 500 个并发线程、10 秒 ramp-up，总库存 1500 座。统计时仅将售空前真实成交请求纳入核心性能口径，售空后的业务拒绝请求不混入成交延迟统计。

| 指标 | 结果 |
| --- | --- |
| 并发线程数 | 500 |
| 总库存 | 1500 座 |
| 真实成交请求数 | 1500 |
| 超卖情况 | 0 超卖 |
| 售空完成窗口 | 约 8.736 秒 |
| 平均完成吞吐 | 约 171.7 orders/s |
| 峰值 1 秒成交吞吐 | 约 268 orders/s |
| P50 响应时间 | 约 758 ms |
| P95 响应时间 | 约 1743 ms |
| P99 响应时间 | 约 2249 ms |

测试结果表明，在单机容器化环境和 500 并发抢票压力下，系统的 Redis Lua 库存扣减、座位锁定和下单链路能够保证成交数量不超过库存，满足防超卖要求。长尾延迟主要来自并发爬坡阶段 Redis 热点路径排队。

## 6 问题分析

| 编号 | 问题 | 影响 | 处理说明 |
| --- | --- | --- | --- |
| R-001 | JMeter 原始响应体未全部保存 | 压测业务分类依赖响应大小和库存推断 | 本报告按原有响应大小和库存口径统计 |
| R-002 | Redis Lua 是下单热点路径 | 高并发阶段会推高 P99 延迟 | 通过分片库存、限流和快速拒绝策略控制影响 |
| R-003 | 售空后拒绝请求数量较大 | 全局延迟指标会受售空后请求影响 | 性能评价按售空前真实成交请求统计 |

## 7 测试结论

本项目已完成覆盖核心公共逻辑、自动选座算法、主要业务链路和故障注入场景的测试，并提供 JMeter 高并发下单压测计划。根据现有测试结果，系统在 1500 库存、500 并发线程的抢票场景下未出现超卖，核心下单链路具备可解释的吞吐和延迟表现。

综合判断，当前提交版本满足课程大作业“提交软件源代码、测试代码、需求规约、概要设计、测试报告和答辩 PPT”的材料要求。
